# Agent 项目面试问题与参考回答

本文基于当前项目实现（agent-only、短期记忆 + 长期记忆 + 工具调用）整理，适合用于技术面试准备。

## 1) 为什么选择“Agent + 记忆 + 工具”架构？
**参考回答：**
我希望系统不仅能回答问题，还能在多轮会话里保持上下文，并在需要时调用外部能力提升准确性。Agent 负责决策，记忆负责个性化连续性，工具负责弥补模型在实时信息和精确计算上的短板。

## 2) 短期记忆和长期记忆分别解决什么问题？
**参考回答：**
短期记忆解决“本次会话连续对话”的上下文问题，依赖 session_id 在进程内维持。长期记忆解决“跨会话保留用户事实”的问题，存储在 SQLite 或 txt 中，用于下次对话检索增强。

## 3) 为什么长期记忆要结构化提取，而不是直接保存原文？
**参考回答：**
直接存原文会引入噪声和冗余，后续检索质量差。结构化提取后可以按 key/value 管理，比如 name、goal、like，方便冲突更新、去重和可解释调试。

## 4) 如何处理冲突记忆（例如姓名变化）？
**参考回答：**
我把字段分为单值和多值。name、goal 是单值字段，新值覆盖旧值；like、dislike 是多值字段，做去重追加。这样既避免旧信息污染，也保留用户偏好的累积信息。

## 5) 为什么选择 SQLite + 混合检索？
**参考回答：**
项目目标是轻量可落地，SQLite 易部署、无外部依赖。检索层采用关键词 + 向量相似度混合，既保留可解释性，又提升语义召回效果，适合中小规模本地 agent 场景。

## 6) 混合检索权重怎么确定？
**参考回答：**
当前使用固定向量权重并做归一化融合。工程上先通过小规模验证集观察命中率和人工可读性，再微调权重。后续可做按 query 类型动态权重，例如实体问答偏关键词、语义改写偏向量。

## 7) 你如何评估检索质量？
**参考回答：**
我会建立小型离线评测集，覆盖姓名、目标、偏好、模糊表达等 query，指标用 Recall@K 和 MRR。上线后增加“答案是否使用了正确记忆”这类任务指标，避免只看检索分。

## 8) 为什么工具要拆分为 tools_*.py 并自动注册？
**参考回答：**
拆分是为了解耦职责，避免单文件膨胀。自动注册降低新增工具时的修改成本和遗漏风险。新增工具只需新增模块并维护 __all__，不用手改中心注册表。

## 9) 自动注册机制的风险是什么？如何控制？
**参考回答：**
风险是误注册、命名冲突、模块导入副作用。控制方式包括：仅扫描 tools_ 前缀、优先使用模块 __all__ 导出、测试校验关键工具可见性，必要时再加白名单和启动时告警。

## 10) Python 执行工具如何保证安全？
**参考回答：**
我做了基础沙箱策略：代码长度限制、执行超时、阻断高风险模式（系统命令/网络/进程相关调用）、输出截断。它不是强隔离沙箱，但在本地 agent 场景下能显著降低滥用风险。

## 11) 如何防止 Agent 滥用工具？
**参考回答：**
通过 system prompt 限定工具使用策略，例如实时事实先搜索、精确计算先跑脚本。同时在工具层保留日志，便于识别“重复调用”模式；下一步可加单轮调用次数上限和缓存。

## 12) 为什么要在 GUI 侧边栏展示工具调用细节？
**参考回答：**
这是可观测性设计。侧边栏能展示 Web Search 和 Python Execution 的过程，帮助我快速判断是“检索失败”“工具失败”还是“推理失败”，减少黑盒调试成本。

## 13) 你做了哪些可观测性建设？
**参考回答：**
当前有搜索日志和 Python 执行日志按轮展示，并记录提取事实和写入事实。这样从用户输入到最终回答的关键链路都可回放，便于定位问题和复盘。

## 14) 多用户并发场景下可能的瓶颈是什么？
**参考回答：**
进程内短期记忆和本地 SQLite 会成为瓶颈。并发下需要把会话状态外置、数据库连接策略优化、工具执行隔离。若规模继续扩大，可引入独立向量服务和任务队列。

## 15) 为什么保留 txt 后端？
**参考回答：**
txt 后端适合演示和极简部署，调试成本低。SQLite 后端用于正式功能，支持结构化存储和混合检索。两个后端共存能兼顾教学、验证和工程演进。

## 16) 如何保证重构不破坏能力？
**参考回答：**
我在重构时先做最小闭环：静态检查 + 关键回归测试（记忆、检索、工具）。例如移除 cot/brief、改自动注册后都跑了回归套件，确保功能一致再提交。

## 17) 这个项目里你做过的关键 trade-off 是什么？
**参考回答：**
我优先选了“可部署性和可解释性”，而不是最高语义能力。比如先用 TF-IDF + SQLite 混合检索，而不是直接上重型 embedding + 向量数据库，保证迭代速度和可维护性。

## 18) 如果上线生产，第一阶段会补哪三件事？
**参考回答：**
第一是更强的执行沙箱（容器/权限隔离）；第二是检索与回答质量评测体系；第三是统一观测平台（结构化日志、trace、告警），让问题可被量化追踪。

## 19) 如何降低幻觉风险？
**参考回答：**
通过“工具优先策略 + 引用可追踪结果 + 回答约束”降低幻觉。对于实时事实优先 web_search，对于计算类问题优先 run_python_code，并在侧边栏保留证据链。

## 20) 如何平衡可解释性与隐私？
**参考回答：**
可解释性需要日志与记忆可见，但隐私要求最小化存储。我的做法是只存必要用户事实、避免原文全量持久化，并在后续增加敏感字段过滤与可删除机制。

## 面试使用建议
1. 每题先用 20 秒讲“目标与选择”。
2. 再用 40 秒讲“实现与权衡”。
3. 最后 20 秒讲“问题与下一步”。

这套回答结构可让你在系统设计面试里更显工程化与主导力。
